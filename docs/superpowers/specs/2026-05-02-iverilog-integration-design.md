# Verilog (Icarus 13.0) — judge0 integration design

**Date:** 2026-05-02
**Author:** gkapatia (with Claude)
**Repos touched:** `Newton-School/compilers`, `Newton-School/judge0`
**Target tags:** `judge0-newton-compiler:0.29`, `newton-judge0:0.66`
**Language id:** `3005` (next free Newton-custom slot after archived `3003`/`3004`)

## Goal

Add Icarus Verilog 13.0 as a new judge0 language so students can submit Verilog
modules (DUTs) and have them simulated against an instructor-supplied
testbench. The simulation output (typically a truth table generated by
`$display` + `$monitor`) is string-matched against the problem's
`expected_output` — same grading model as every other judge0 language.

## Submission model

| field             | content                                                                                                |
| ----------------- | ------------------------------------------------------------------------------------------------------ |
| `source_code`     | Student's DUT module(s) — e.g. `module d_flipflop(...); ... endmodule`. Saved to `main.v` in the box.  |
| `stdin`           | Instructor's testbench — e.g. `module tb_d_flipflop_simple; ... endmodule`. Captured to `tb.v`.        |
| `expected_output` | Exact stdout string produced by `vvp` after the `$finish` line is filtered out.                        |

This uses `stdin` (not `additional_files`) as the testbench transport because
the platform-side path of least resistance is to reuse the existing
testbench-delivery mechanism. Trade-off: the testbench cannot ALSO read runtime
stdin during simulation (`$fgetc(STDIN)` etc.) since `cat > tb.v` consumes it.
For the standard "testbench drives fixed stimulus, prints results" pattern this
is fine.

## Compile / run split (option B)

judge0's `compile_cmd` runs **before** stdin is piped — stdin is only available
to `run_cmd`. We therefore can't link both files at compile time. Instead:

- **`compile_cmd`** parse-checks `main.v` *alone* with `iverilog -tnull`. Pure
  syntax errors in the student's DUT surface as **Compilation Error** in
  judge0's status bucket.
- **`run_cmd`** captures stdin to `tb.v`, then full-elaborates `main.v + tb.v`,
  then runs `vvp`, with the trailing `$finish` line filtered out.

Bucket implications:

| Failure                                        | Bucket           |
| ---------------------------------------------- | ---------------- |
| Syntax error in `main.v`                       | Compile Error    |
| Syntax error in testbench                      | Runtime Error    |
| Port-mismatch / undeclared module elaboration  | Runtime Error    |
| `$display` output != `expected_output`         | Wrong Answer     |
| Successful match                               | Accepted         |

Testbench-side syntax errors landing in the Runtime bucket is a known
misclassification but in practice is acceptable: the testbench is
problem-author-supplied (caught once during problem authoring, not in student
traffic), and the iverilog stderr message is preserved either way.

## Toolchain placement (compilers repo)

Append a new bottom-tier layer to `NewtonDockerFiles/NewtonDockerfile-v2`. Per
the layer-ordering rule in `compilers/CLAUDE.md`, new languages append at the
bottom — never insert mid-file.

```dockerfile
# Tier N — Verilog (Icarus 13.0)
ENV IVERILOG_VERSION 13.0
RUN set -xe \
    && apt-get update && apt-get install -y --no-install-recommends \
         autoconf gperf flex bison make \
    && cd /tmp \
    && curl -fsSL https://github.com/steveicarus/iverilog/archive/refs/tags/v${IVERILOG_VERSION}.tar.gz | tar xz \
    && cd iverilog-${IVERILOG_VERSION} \
    && sh autoconf.sh \
    && ./configure --prefix=/usr/local/iverilog-${IVERILOG_VERSION} \
    && make -j"$(nproc)" \
    && make install \
    && cd / \
    && rm -rf /tmp/iverilog-${IVERILOG_VERSION} \
    && apt-get purge -y autoconf gperf flex bison \
    && apt-get autoremove -y && apt-get clean && rm -rf /var/lib/apt/lists/*
```

- Install path: `/usr/local/iverilog-13.0/bin/{iverilog,vvp}`.
- Multi-arch: pure C/C++ source. Builds cleanly on amd64 and arm64 with no
  `TARGETARCH` branching.
- Build time: ~5 min on a c6i.4xlarge (one-time, bottom of the cascade).
- Build deps (`autoconf gperf flex bison`) are installed and purged in the same
  RUN, so they don't survive into the final image. `make` stays because other
  layers may rely on it.

## Smoke test addition (compilers repo)

Append to `compilers/bin/newton-test`:

```bash
mkdir -p /tmp/verilog && cd /tmp/verilog
cat > hello.v <<'EOF'
module testbench;
  initial begin
    $display("Hello, World!");
    $finish(0);
  end
endmodule
EOF
/usr/local/iverilog-13.0/bin/iverilog -g2012 -o hello.vvp hello.v \
  && /usr/local/iverilog-13.0/bin/vvp hello.vvp | grep -q "Hello, World!" \
  && echo "PASS: Verilog (Icarus 13.0)"
```

(The smoke test runs iverilog directly without judge0's stdin-routing, so it
uses a single self-contained `.v` file. `$finish(0)` keeps the smoke output

## Language entry (judge0 repo)

Append to `db/languages/active.rb`:

```ruby
{
  id: 3005,
  name: "Verilog (Icarus 13.0)",
  is_archived: false,
  source_file: "main.v",
  compile_cmd: "/usr/local/iverilog-13.0/bin/iverilog -g2012 -tnull %s main.v",
  run_cmd: "/bin/cat > tb.v && /usr/local/iverilog-13.0/bin/iverilog -g2012 -o sim.vvp main.v tb.v && /usr/local/iverilog-13.0/bin/vvp sim.vvp | grep -v 'finish called at '"
}
```

`compile_cmd` breakdown:

- `iverilog -g2012` — SystemVerilog 2012 mode. Strict superset of Verilog-2005,
  enables `logic`/`bit`/`always_ff`/`class`/`interface` for advanced courses
  while keeping classic-style code working.
- `-tnull` — parse-only target; emits no output. Exits 0 on parse-success,
  non-zero on parse-error. Verified on iverilog 13.0.
- `%s main.v` — judge0 substitutes any flags from `submission.compiler_options`
  in for `%s`.

`run_cmd` breakdown:

- `cat > tb.v` — captures stdin (testbench source) to a sibling file in the
  isolate box.
- `iverilog -g2012 -o sim.vvp main.v tb.v` — full elaboration. No `-s <module>`
  flag: iverilog auto-detects the top-level module (any module not instantiated
  by another). Problem authors are free to name their testbench module anything
  (`tb_adder`, `tb_d_flipflop_simple`, etc.).
- `vvp sim.vvp | grep -v 'finish called at '` — runs the simulation and
  filters out the `tb.v:N: $finish called at T (1s)` epilogue that vvp writes
  to stdout. Without this filter every problem's `expected_output` would have
  to include this volatile line (file:line + sim time both vary), which would
  break grading. Pattern uses `'finish called at '` rather than `'\$finish'`
  to avoid double-escape gymnastics across the Ruby-string → bash-c boundary
  in IsolateJob — `$` doesn't need escaping if it's not in the pattern. The
  string `finish called at ` is unambiguous (a `$display` containing those
  exact words is implausible in a grading testbench).

## Author conventions (problem-side)

To be documented in the problem-authoring guide:

1. **Capture expected output by running the testbench locally** — don't
   hand-trace `$monitor`. `$monitor` consolidates same-time-step changes into a
   single row (IEEE 1364-2005 §17.1), so hand-traced "before/after posedge"
   row pairs will not appear. To split before/after rows, use explicit
   `$display` calls with delays.
2. **Testbench should not use `$dumpfile`/`$dumpvars`** — VCD dumps print a
   `VCD info: ...` line to stdout that would have to be matched in
   `expected_output`. (Filtering this line out as well is a possible future
   addition; for now, document the constraint.)
3. **Submission should contain the DUT module(s) only** — no extra unconnected
   top-level modules in `main.v`. iverilog's auto-detection runs every
   unconnected module as a top-level in parallel, which would interleave
   output with the testbench.

### Phase 3 — waveform delivery (separate spec, future)

Verilog learning is half-blind without waveform inspection. iverilog already
generates `.vcd` files when the testbench calls `$dumpfile`/`$dumpvars`, but
judge0 today has no mechanism to return file artifacts beyond stdout/stderr.
Three design options to evaluate when this phase opens:

- **A) Base64-encode `dump.vcd` into stdout with a marker** (e.g.
  `===VCD===\n<base64>\n===END===`). Platform UI parses out the marker,
  decodes, renders via `wavedrom` / `vcd.js`. Zero judge0-Rails changes —
  just a `run_cmd` extension. Trade: stdout bloat (50–500 KB per VCD), and
  the grader must split before the marker.
- **B) Capture box artifacts in `IsolateJob`** — add an `artifacts` field
  to the language definition listing files to copy out before
  `isolate --cleanup`, return them base64'd in a new submission response
  field. Cleaner separation; multi-day Rails change with a schema migration.
- **C) S3 / external artifact store** — `IsolateJob` uploads the VCD,
  returns a presigned URL. Heavier infra, scales further but more moving
  parts.

A new spec under `docs/superpowers/specs/` should evaluate these against the
platform's UI plans before committing. Recommendation pending Phase 3
brainstorm. Until then, the spec's author conventions (no `$dumpfile`)
remain in force; testbenches that include `$dumpvars` will emit the VCD
into the box where it's wiped on cleanup, with the `VCD info:` line
polluting graded stdout.

### Production redeploy

Standard operational followup, not feature-specific work. The AL2 → AL2023
karpenter NodeClass migration is the gating prerequisite (already tracked
separately) for any image bump, including this one — without it,
`docker-entrypoint.sh` falls back to rlimit mode and the cgroup-v2 memory/CPU
semantics stop working. Once that's flipped, deploying `newton-judge0:0.66`
follows the same path as previous bumps. No Verilog-specific prod
prerequisite.

## Risks and open items

- **`grep -v` exit code:** if vvp produces *only* the `$finish` line (e.g., a
  testbench that does nothing but `$finish`), `grep -v` exits 1 because no
  lines remain. judge0 may interpret that as a failure. Verify in dev; if it
  bites, switch the filter to `sed '/: \$finish called at /d'` (always exits 0
  on stream end).
- **Multi-module student submissions:** iverilog accepts multiple modules in
  one `.v` file. If a student writes a helper module + a DUT, both compile;
  the testbench instantiates the DUT and helpers come along for the ride.
  This is fine and matches how Verilog is normally written.
- **`compiler_options` field:** judge0 lets users pass extra flags via the
  `compiler_options` API field, substituted into `%s`. For Verilog this would
  add flags to `iverilog` (e.g., `-Wall`). Run-time flags (e.g., to vvp)
  cannot be passed this way. Document this limitation if/when it comes up.

## Out of scope (this spec)

- **Waveform delivery.** Promoted to Phase 3 — see Operational sequence above.
  Has its own design space (options A/B/C) and warrants a separate spec.
- **Multi-file submissions via `additional_files`.** Could be a clean way to
  deliver the testbench, but switches the platform-side data flow; defer
  until/unless we have a reason to revisit.
- **A separate `id` for SystemVerilog-only mode.** Single id with `-g2012`
  covers both styles per the brainstorm decision.
- **Verilator support.** Different toolchain (Verilog → C++ → native binary),
  requires C++ testbench harnesses, doesn't run pure-Verilog testbenches.
  Worth a separate language id if/when an advanced verification track exists.
